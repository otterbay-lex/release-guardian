# 工程质量规则

## 目标

判断项目是否能被他人可靠地安装、构建、运行和维护。工程问题与合规问题同等重要。

## 检查顺序

1. 从 manifest、README、CI 和脚本识别技术栈与官方命令。
2. 运行行为清晰、不会修改生产状态的已有 lint、typecheck、test、build。
3. 记录命令、退出码和失败摘要；不要只写“测试失败”。
4. 检查依赖锁文件是否与技术栈匹配，避免擅自生成锁文件。
5. 查找硬编码本机路径、操作系统假设、调试开关和危险默认值。
6. 验证 README 描述的最小核心流程；不能实际验证时标记“需要人工确认”。

## 严重程度

- 阻断：无法安装或构建；核心流程不可用；发布入口不存在；明确的数据破坏错误。
- 重要警告：测试失败；类型检查失败；锁文件缺失或不一致；明显的平台兼容问题。
- 建议：缺少测试、lint 或类型检查，但已有核心流程证据。缺少测试不能单独自动阻断发布。

## 证据要求

写明命令、文件、退出码和最短必要错误摘要。不要把未运行的命令写成已通过。
