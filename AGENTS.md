# AGENTS.md

## Project Context

Release Guardian 是一个通用型 AI 发布质量检查 Skill，服务初级开发者和缺乏合规经验的产品负责人。项目当前处于 v2 设计阶段，工程质量与合规风险必须同等重视。

当前不实现自动部署、云平台代操作或正式法律意见。真正的公开发布始终需要用户单独授权。

## Working Principles

- 修改前阅读 `README.md`、`ARCHITECTURE.md`、`TASKS.md` 和相关设计文档。
- 一次只处理一个明确任务，不主动扩大范围。
- 保留现有 Release Guardian 的有效能力，在其基础上深化。
- 优先由 AI 完成读取、搜索、运行、比较和验证，减少用户手工操作。
- 未经确认，不安装依赖、不执行破坏性操作、不修改外部平台、不发布。
- 不重写无关文件，不进行无关重构。
- 使用简单、清晰、适合初级开发者理解的中文解释。
- 事实、推断和待人工确认事项必须分开表达。

## Code Style

- 保持主 `SKILL.md` 精简，把详细规则放入按需读取的 `references/`。
- 脚本优先使用 Python 标准库，避免不必要依赖。
- 函数职责单一，命名清晰；文件接近 300 行时评估拆分。
- 检查规则使用稳定标识，并记录证据、可信度、适用条件和复核方式。
- 不在日志或报告中输出完整密钥、个人信息或认证数据。

## Engineering Habits

- 检查现有结构并复用已有模式。
- 新增依赖前解释必要性并获得确认。
- 修改后运行适用的格式、测试和静态检查。
- 没有自动化测试时提供明确的人工验证步骤。
- 高风险发现必须经过上下文判断，不能仅凭关键词定性。
- 检查源码时同时确认最终发布物，避免对象不一致。

## Design And Planning

- 实施前阅读 `docs/superpowers/specs/2026-07-15-release-guardian-v2-design.md`。
- 设计经用户确认后，先创建详细实施计划再修改 Skill。
- 大型或不确定改动优先使用 Worktree。

## Output Format

每次完成任务后说明：

1. 修改了哪些文件。
2. 实现了什么。
3. 如何验证。
4. 是否有风险或后续建议。
