# Release Guardian v2 设计

日期：2026-07-15

## 1. 产品定位

Release Guardian v2 是面向初级开发者和非合规专业人士的通用型 AI 发布质量守门员。它在代码仓库、网页、App、API、软件包、AI 应用、Codex Skill、文档、数据集或最终制品公开前，主动完成工程、发布、安全、合规、知识产权和运维复核。

它不是单纯的合规扫描器。它回答的核心问题是：这个项目现在是否适合交给别人使用、公开发布或部署上线？

它提供发布前初步审查，不代替全面安全审计或正式法律意见。

## 2. 目标与成功标准

### 目标

- 保留原版 Secrets、Privacy、Git Ignore、Demo Data、License、Version 和 Security 能力。
- 补齐构建、测试、依赖、制品、升级、回滚与维护能力检查。
- 以中国大陆合规为主，并兼顾全球通用隐私、版权和开源要求。
- 让 AI 主动完成能够读取、搜索、运行、比较和验证的工作。
- 用初级开发者能理解的语言解释风险和修复步骤。
- 先检查和报告，用户确认修复批次后再修改。

### 成功标准

- 能识别常见项目和组合类型，并选择适用规则。
- 工程问题不会被合规风险掩盖，合规问题也不会被构建成功掩盖。
- 高风险问题具有证据、上下文、可信度、适用条件和复核方式。
- 不把占位 Key、虚构数据或字段名轻率判定为真实泄露。
- 明确区分已验证、发现问题、需要人工确认和不适用。
- 检查对象与最终发布物一致；无法一致时不得给出“可以发布”。
- 未经授权不安装依赖、不修改外部平台、不推送、不部署、不发布。

## 3. 工作模式

采用两阶段检查。

### 快速体检

用于日常准备，优先运行项目已有的构建、测试、lint 或类型检查，扫描密钥、隐私数据、本机路径、调试文件，并检查 README、许可证、版本、锁文件、`.gitignore`、Git 状态和实际待发布内容。

快速体检不安装新依赖，也不自动修改文件。

### 深度检查

用于正式发布前，进一步验证干净环境安装、核心使用流程、依赖与供应链、第三方授权、最终制品、CI、升级、迁移、备份、回滚、监控，以及适用的隐私、AI 和中国合规事项。

耗时、联网、产生费用或改变环境的操作需要用户确认。

## 4. 六大检查支柱

| 支柱 | 权重 | 主要内容 |
|---|---:|---|
| 工程质量 | 25 | 构建、测试、lint、类型检查、锁文件、兼容性、错误处理 |
| 发布完整性 | 20 | 版本、变更记录、安装说明、配置示例、发布物、升级和回滚 |
| 安全与供应链 | 20 | 密钥、权限、漏洞、依赖来源、危险脚本、最小权限 |
| 隐私与合规 | 15 | 个人信息、日志、Cookie、SDK、数据处理和中国监管提示 |
| 知识产权 | 10 | 开源许可证、第三方代码、图片、字体、模型和数据集授权 |
| 运维与可恢复性 | 10 | 监控、限流、费用、备份、恢复和事故响应 |

分数用于帮助理解，不覆盖硬阻断规则。

## 5. 项目识别与规则组合

所有发布对象先执行通用检查，再按文件、清单和目录识别专项类型：

- GitHub 开源仓库。
- 网页或前端。
- API 或后端服务。
- 桌面或移动 App。
- 命令行工具或软件包。
- AI 应用。
- Codex 或 Agent Skill。
- 文档、模板或数据集。
- ZIP、安装包或镜像。

组合项目应叠加规则。例如 AI Web 应用同时使用网页、API、AI 和通用规则，不要求用户掌握技术分类。

## 6. Agent 优先协作

在用户授权的项目范围内，AI 主动完成项目识别、文档和配置阅读、Git 检查、已有质量命令执行、敏感信息扫描、依赖分析、制品比较、证据收集、评分和复核。

AI 应先调查再提问：先从现有材料推断，无法确认时采用明确的保守假设，只有答案会显著改变范围或涉及外部权限时才提出简短问题。

多 Agent 可把工程、安全、合规和知识产权检查并行分工，但只是可选增强。主 Agent 负责去重、冲突处理和统一结论；单个 AI 也必须能完成检查。

## 7. 修复边界

采用“报告 → 修复批次 → 一次确认 → AI 修改 → 重新验证”的流程。

- 可安全自动修复：补充忽略规则、修正示例路径、增加占位环境变量示例。
- 确认后修复：构建配置、依赖升级、许可证、发布脚本和文档。
- 仅提供指导：密钥轮换、生产权限、数据删除、备案和外部平台设置。
- 专家介入：真实数据泄露、复杂许可证冲突、高危漏洞和监管许可不明。

安装或升级依赖、删除数据、产生费用、修改外部账号、推送、部署和正式发布必须单独确认。

## 8. 硬阻断规则

以下问题阻止“可以发布”结论：

- 真实或高度疑似真实的密钥、密码、Cookie 或私钥。
- 真实个人信息、客户数据、案件材料或生产数据库进入发布物。
- 项目无法构建、安装或完成核心使用流程。
- 发布物与实际检查对象不一致。
- 已确认的高危依赖或恶意供应链风险。
- 明显缺失必要授权的第三方资源。
- 危险默认密码、公开管理接口或严重越权风险。
- 可能破坏数据的升级没有备份、迁移或回滚方案。
- 明知需要许可、备案或用户授权却完全没有处理。
- 检查范围不足，无法判断实际发布物。

## 9. 结论与报告

发布结论分为：可以发布、有条件发布、暂不建议发布、禁止自动发布。

报告采用两层结构。第一层面向小白，显示结论、总分、阻断项、重要警告和 AI 可以协助的修复；第二层保存证据化明细。

每项发现包括：稳定标识、严重程度、证据位置、可信度、可能后果、修复方式、AI 可执行范围、用户动作和复核方式。

报告必须披露检查路径、排除目录、执行命令、未完成检查、源码与制品关系，以及外部设置是否实际验证。

固定声明：本次检查基于可访问文件、配置和运行结果，不构成全面安全审计或正式法律意见；未发现问题不代表不存在风险。

## 10. 误报控制

- 占位 Key 与真实密钥分开处理。
- 示例邮箱和测试号码结合上下文判断。
- 许可证风险结合分发方式和项目许可证判断。
- 字段名不等于真实个人信息。
- 没有发现漏洞不等于依赖绝对安全。
- 缺少隐私政策是否阻断取决于产品是否处理用户数据。

高风险结论不能只依赖关键词命中。证据不足时使用“疑似”或“需要确认”，不作绝对法律结论。

## 11. 模块架构

主 `SKILL.md` 负责触发、对象确认、模式选择、规则路由、权限边界和报告汇总。详细规则拆入：

- `engineering.md`
- `release-readiness.md`
- `security-supply-chain.md`
- `privacy-china.md`
- `open-source-ip.md`
- `operations.md`
- `project-profiles.md`
- `reporting.md`

可选 `release_scan.py` 使用 Python 标准库执行确定性扫描。脚本不可用时，AI 仍可使用常用命令和文件检查完成基础审查。

## 12. 测试设计

第一轮测试覆盖：

1. 含本机路径、许可证缺失和前端 Key 的静态网站。
2. 构建成功但测试失败、无锁文件且版本过期的 Node 项目。
3. 保存聊天记录但数据处理说明不足的 AI 应用。
4. 含脚本权限、绝对路径和外部依赖问题的 Codex Skill。
5. 仅含占位 Key 和虚构数据的假阳性样本。
6. 源码干净但 ZIP 残留 `.env` 或调试数据库的制品不一致样本。

评价内容包括项目类型识别、阻断项召回、明显误报、解释可读性、AI 与用户动作边界、证据谨慎性以及是否未经确认修改或发布。

## 13. 非目标

- 不承诺覆盖所有司法辖区和行业监管要求。
- 不以付费工具或联网服务作为基础检查前提。
- 不在检查通过后自动执行公开发布。
- 不用单一总分替代专业判断。
- 不要求用户手工完成 AI 可以安全执行的复核工作。
